  • A l'heure et aux jours de la semaine de votre choix.
  • A la demande.
  • A la demande avec une temporisation.
  • Nombre illimité d'actions.
  • Modification et gestion vocale.
  • Gère le multi-rooms.

Création d'un scénario

La création d'un scénario se fait dans le fichier scenariz.xml.
Chaque action du scénario est une ligne à vocaliser.
La clé principale:
out.action.command="ScenarizCron" : Doit toujours être présente.
Les autres clés d'une action:
out.action.clients : Définition des clients sarah qui exécutent le programme. Si il y a plusieurs clients, utilisez le caractère '|' pour les séparer ou 'All' pour tous les clients. Non obligatoire ("SARAH1" par défaut).
out.action.program : Le nom du scénario. Toutes les actions d'un scénario doivent avoir le même nom de programme. Non obligatoire ("Général"par défaut).
out.action.name : Le nom de l'action dans le programme. Toutes les actions d'un scénario doivent avoir un nom différent. Obligatoire.
out.action.order : Ordre de l'action si plusieurs actions sont à exécuter à la même heure. Non obligatoire ("1" par défaut).
out.action.tempo : Temporisation après la prochaine action si plusieurs actions sont à exécuter à la même heure. En milli-secondes. Non obligatoire ("1000" par défaut).
out.action.exec : L'action est à exécuter ('true') ou à ignorer ('false'). Non obligatoire ('true' par défaut).
out.action.plug : Le nom exact du module à exécuter. Obligatoire.
out.action.key : La commande et les clés du module. Non obligatoire (aucune par défaut).
Format: clés 1=valeur 1~clé 2=valeur 2~clé 3=valeur 3
Exemple pour le module météo (aucune commande, juste des clés pour la ville et la date):
out.action.key="zip=315550~date=0"
Exemple pour le module SonosPlayer pour démarrer une musique (clé "command" pour la commande dans le module, clé "type" pour le type de média et clé "title" pour le titre):
out.action.key="command=set_media~type=favorite~title=Mes Préférées"
out.action.start : L'heure et les jours d'exécution de l'action. Obligatoire.
Format: heure:minutes-7 chiffres 0 ou 1 représentant les 7 jours de la semaine commencant le lundi. 1: le jour est exécuté. 0 le jour est ignoré.
Exemple: 07:45-1111100 exécute l'action à 07h45 les 5 premiers jours de la semaine
out.action.ttsCron : Un texte à vocaliser si besoin pour l'action. Non obligatoire (aucun par défaut).
Le caractère spécial %s est automatiquement remplacé par le callback tts du module dans la phrase.
Exemple avec le module météo qui retourne la météo dans un callback tts:
out.action.ttsCron="La %s. Passe une bonne journée."

Exemple de création d'un scénario

Exemple d'un scénario de réveil en multi-rooms exécuté les jours ouvrés (lundi à vendredi).
Les 3 premières actions sont exécutées à 07h45 dans l'ordre.
  • 1: Une phrase de réveil.
  • 2: La météo.
  • 3: Démarre ma musique.
  • 4: Monte le son à 7h55.
  • 5: Une phrase de rappel à 08h10.
  • 6: Monte le son à 08h10.

1: La phrase de réveil : out.action.ttsCron="Bonjour Stéphane. Il est %s, il est l'heure de te réveiller."
scenariz a une commande "setTime" qui retourne l'heure dans un callback tts.
le %s est donc remplacé par l'heure si vous définissez out.action.plug="scenariz" et out.action.key="command=setTime"

Il est %s, il est l'heure de te réveiller.";out.action.key="command=setTime"‹/tag›‹/item›
Pour cette exemple, j'ai un multi-rooms. Pour faire en sorte que le réveil s'exécute dans la pièce de mon choix, je définis dans out.action.client le nom du client Sarah de la chambre qui est "CHAMBRE" où l'action sera exécutée.
vous pouvez voir aussi une tempo de 6 secondes pour l'action suivante, ce qui laisse le temps à Sarah de vocaliser la phrase avant de démarrer l'action 2 "météo".

2: La Météo
Nom du plugin: meteo.
2 clés "zip" et "date" sont à définir.
Pour la qualité du message, j'encapsule le callback tts de la météo dans une phrase définie par la clé out.action.ttsCron (voir ci-dessus les explications sur les clés). Ce qui donnera:

Passe une bonne journée.";out.action.key="zip=315550~date=0"‹/tag›‹/item›
Vous pouvez voir une temporisation de 25 secondes ainsi que la clé out.action.order qui passe à "2" puisque les actions sont exécutées à la même heure.

Création et exécution d'un scénario à la demande

Un scénario à la demande est généralement créé pour réaliser une série d'actions immédiates ou avec une temporisation.
Par exemples, un scénario de démonstration à exécution immédiate pour épater vos amis, des actions à réaliser lorsque vous partez de chez vous (exécution avec un léger différé) ou peut-être encore lorsque vous allez vous coucher.
Il peut être aussi appelé par un autre plugin et effectué des actions en fonction de critères, etc...

La création d'un scénario à la demande est identique à un scénario défini pour une heure et un jour de la semaine, d'ailleurs celui-ci peut être exécuté à la demande en plus de l'heure et le jour défini.

2 clés sont définies pour les scénarios à la demande: La clé principale:
out.action.command="ExecCron" : Doit toujours être présente. Commande d'exécution d'un scénario à la demande.
L'autre clé d'une action:
out.action.program : Doit toujours être présente. Le nom du scénario (clé out.action.program) à exécuter.
out.action.timeout : Le timeout d'exécution. Non obligatoire. Si non défini, le scénario est exécuté immédiatement.
A noter que dans le fichier scenariz.xml, en bas du fichier, des délais de temporisation de 5 minutes en 5 minutes sont présents.

A Savoir:
  • Comme la clé out.action.start est obligatoire à la création des actions (voir ci-dessus), mettez n'importe quelle heure et définition de jours pour l'éxécution. Ces valeurs seront automatiqument changées pour toutes les actions lors de l'appel du scénario pour l'heure courante et/ou avec la temporisation.
  • La clé out.action.exec est uniquement utilisée pour les scénarios à heure et jours définis. Elle est non obligatoire mais est définie à 'true' si non présente. Et donc le scénario sera exécuté à l'heure et aux jours même si celui-ci est un scénario à exécution immédiate. Pour éviter qu'il soit exécuté comme un scénario programmé, pour les scénarios à exécution immédiate, définissez toujours cette clé à 'false'.

Exemple de vocalisation:
Sarah fais nous une petite démo
Exécutera le scénario tout de suite
Sarah fais nous une petite démo dans cinq minutes
Exécutera le scénario le multiple inférieur de 5 avant la temporisation.
Par exemple, si "dans 5 minutes" défini l'heure d'exécution à 17h28, il sera exécuté à 17h25.

Modification d'un scénario

La modification d'un scénario peut être effectuée de 2 façons:
  • 1: Soit en modifiant l'action dans le fichier scenariz.xml et en la rééxécutant.
  • 2: Soit en utilisant l'ordre vocal "Modification des Programmes"

1: Modifier l'action dans le fichier scenariz.xml
Il suffit juste de modifier les clés que vous voulez et de rééxécuter l'ordre vocale, par exemple, si vous voulez modifier l'heure, ou une phrase, le plugin, une clé...
2 clés qui représentent "la clé unique" de l'action ne doivent JAMAIS être modifiées:
  • Le nom de l'action (la clé out.action.name)
  • Le programme (la clé out.action.program)
sinon cela sera vu comme une nouvelle action.

2: L'ordre vocal "Modification des Programmes"
Sarah récupère la liste des scénarios et vous propose de les modifier.
Vous pouvez répondre:
  • Qu'est ce que je peux dire
    • -> Rappel des actions possibles (ci-dessous)
  • Oui modifie
    • -> Pour modifier le scénario
  • Suivant
    • -> Pour passer au suivant, si il en existe
  • Inverse le sens
    • -> Pour revenir en arrière ou aller en avant dans la liste des scénarios, si il en existe
  • Quel est le sens
    • -> Pour connaitre le sens, en arrière ou en avant dans la liste
  • terminé
    • -> Pour terminer et sortir du mode "gestion de programme"
Modification
Vous pouvez répondre:
  • Qu'est ce que je peux dire
    • -> Rappel des actions possibles (ci-dessous)
  • L'état
    • -> Pour afficher l'état du scénario, vous donnera le(s) client(s) qui exécute(nt) le programme, le nombre d'actions, le début et la fin du programme ainsi que les jours d'exécution.
  • Activer
    • -> passe la clé out.action.exec à 'true' pour toutes les actions du scénario et le réactive.
    -> Note: Attention aux programmes à exécution immédiate. Voir le chapitre ci-dessus.
  • Désactiver
    • -> passe la clé out.action.exec à 'false' pour toutes les actions du scénario et le désactive.
  • supprime le programme
    • -> Supprime le scénario et toutes ses actions.
  • Les minutes
    • -> Pour modifier les minutes de début du scénario. La première action du premier ordre est utilisée mais toutes les minutes de toutes les actions sont modifiées relativement.
  • L'heure
    • -> Pour modifier l'heure de début du scénario. La première action du premier ordre est utilisée mais toutes les heures de toutes les actions sont modifiées relativement.
  • les jours
    • -> Pour modifier les jours du scénario. Toutes les actions sont modifiées.
  • Terminé
    • -> Pour sauvegarder les modifications.
  • Annuler
    • -> Pour quitter sans sauvegarder les modifications.
Les minutes et Heures
Identique dans les ordres pour les minutes et les heures.
Vous pouvez répondre:
  • Qu'est ce que je peux dire
    • -> Rappel des actions possibles (ci-dessous)
  • Baisse
    • -> Pour réduire de 5 minutes ou 1 heure
  • Baisse beaucoup
    • -> Pour réduire de 15 minutes ou 5 heures
  • Augmente
    • -> Pour augmenter de 5 minutes ou 1 heure
  • Augmente beaucoup
    • -> Pour augmenter de 15 minutes ou 5 heures
  • Terminé
    • -> Pour sauvegarder les modifications.
  • Annuler
    • -> Pour quitter sans sauvegarder les modifications.
Les jours
Vous pouvez répondre:
  • Qu'est ce que je peux dire
    • -> Rappel des actions possibles (ci-dessous)
  • Activer
    • -> Pour activer les jours (la phrase de Sarah vous montre si elle active ou désactive)
  • Désactiver
    • -> Pour désactiver les jours (la phrase de Sarah vous montre si elle active ou désactive)
  • Semaine de travail
    • -> Pour définir du lundi au vendredi
  • Toute la semaine
    • -> Pour définir toute la semaine
  • Le Lundi
    • -> Pour activer/désactiver le Lundi
  • Le Mardi
    • -> Pour activer/désactiver le Mardi
  • Le Mercredi
    • -> Pour activer/désactiver le Mercredi
  • Et même chose pour tous les jours jusqu'au Dimanche
  • Terminé
    • -> Pour sauvegarder les modifications.
  • Annuler
    • -> Pour quitter sans sauvegarder les modifications.

Tests

La création d'un scénario necessite des tests.
Pour effectuer des tests, une bonne pratique consiste à exécuter le scénario comme un scénario à la demande.
Créez donc une ligne "ExecCron" dans le fichier scenariz.xml avec le nom du programme pour le tester.

Configuration

Le Cron s'active toutes les 5 minutes.
Si une heure est définie en non multiple de 5 minutes, le module activera automatiquement l'action pour le multiple inférieur.

A noter qu'il est possible de modifier l'activation toutes les 5 minutes. Ouvrez le fichier scenarizdb.js, afficher la fonction "istime" et suivez l'aide de la fonction. N'oubliez pas de modifier la valeur du cron.

Le module utilise 2 bases nedb localisée dans le répertoire lib/db et nommées:
  • Scenariz.db pour la base principale.
    • Cette base est "LA" base de sauvegarde de tous les scénarios.
  • ScenariznoCron.db pour la base secondaire.
    • Cette base est la base de sauvegarde des actions supplémentaires à effectuer pour un scénario à la demande. Comme les actions d'un scénario ne sont pas toutes définies à la même heure, les actions suivantes à exécuter sont stockées temporairement dans cette base.
    Après leurs exécutions, elles sont automatiquement supprimées.
    Il peut arriver que certaines actions ne soient pas supprimer, par exemple si en cours d'exécution d'un scénario, Sarah est arrétée et n'est redémarrée que après l'heure d'éxécution d'une action de scénario, l'action restera alors dans la base.
Ces bases sont des fichiers textes que vous pouvez modifier dans un éditeur de texte.
Par exemple, pour supprimer une action, changer une heure... vous devez juste respecter la syntaxe.
Vous pouvez aussi supprimer les fichiers. Il seront automatiquement recréés.

Il existe un mode debug que vous pouvez activer en modifiant la valeur "debug" du fichier prop à "true".
Ce module est multi-langages.
Pour modifier la langue, changer la valeur "language" du fichier prop avec le nom d'un fichier lang du répertoire lib/lang (le nom du fichier est la valeur à positionner).
Pour en créer un, copier le fichier FR_fr.js et nommer-le par exemple EN_en.js et modifier toutes les valeurs textes.
Tous les messages sont aussi par définition personnalisables. Ouvrez le fichier FR_fr.js et modifiez les valeurs textes des messages à votre convenance.
